## 1. 저장소 포크하기

**포크**란 원본 저장소(다른 사람 계정)를 그대로 복사해 **내 GitHub 계정 아래**에 새 저장소를 만드는 것.

1. GitHub에서 FoliumOnline **원본 저장소** 페이지로 감.
2. 오른쪽 상단 **Fork** 버튼 클릭.
3. "Create a fork"에서 Owner를 **내 계정**으로 두고, 저장소 이름은 그대로 두거나 원하는 대로 바꾼 뒤 **Create fork** 클릭.
4. 끝나면 내 계정 아래에 `FoliumOnline`(또는 지정한 이름) 저장소가 생김. 이게 **내 포크(fork)**.

이후 작업은 **반드시 내 포크 저장소**에서 함. 원본 저장소에 직접 푸시할 권한은 없음.

## 2. 원본(origin) 변경 사항을 내 포크에 가져오기

원본 저장소(예: `owner/FoliumOnline`)가 업데이트되었을 때, 그 변경 사항을 **내 포크**에도 반영하고 싶다면 아래 순서대로 함.

### 2.1 원본을 "리모트"로 추가하기 (한 번만 하면 됨)

내 컴퓨터에서 프로젝트 폴더를 연 터미널(또는 명령 프롬프트)에서:

```bash
git remote add upstream https://github.com/원본소유자/FoliumOnline.git
```

`원본소유자`는 실제 원본 저장소 주인 계정 이름으로 바꿈.  
이렇게 하면 "내 포크"는 `origin`, "원본"은 `upstream`이라는 이름으로 구분됨.

### 2.2 원본 최신 내용 가져오기

```bash
git fetch upstream
```

그다음 **지금 쓰는 브랜치**(예: `main` 또는 `develop`)에 원본 내용을 합침:

```bash
git checkout main
git merge upstream/main
```

원본이 `develop` 브랜치를 메인으로 쓴다면 `upstream/main` 대신 `upstream/develop`으로 바꿈.

### 2.3 내 포크(GitHub)에도 반영하기

로컬에서 합친 뒤, 내 포크 저장소에 푸시함:

```bash
git push origin main
```

이후 GitHub에서 내 포크 페이지를 새로고침하면 원본과 동일한 최신 커밋이 반영되어 있음.

## 3. 충돌(merge conflict) 해결하기

`git merge upstream/main`(또는 다른 브랜치) 실행 시 **Merge conflict** 메시지가 나오면, 같은 파일을 원본과 내가 둘 다 수정했을 때 발생함.

### 3.1 충돌이 났다는 것 확인하기

터미널에 다음 같은 문구가 보임:

- `CONFLICT (content): Merge conflict in ...`
- `Automatic merge failed; fix conflicts and then commit the result.`

### 3.2 어떤 파일이 충돌했는지 보기

```bash
git status
```

"Unmerged paths" 또는 "both modified" 로 나오는 파일이 충돌한 파일.

### 3.3 파일 안에서 충돌 표시 이해하기

충돌한 파일을 열면 대략 다음처럼 표시되어 있음:

```
<<<<<<< HEAD
(내가 수정한 내용)
=======
(원본/다른 쪽에서 수정한 내용)
>>>>>>> upstream/main
```

- `<<<<<<< HEAD` ~ `=======` : **내 쪽** 내용.
- `=======` ~ `>>>>>>> upstream/main` : **가져오는 쪽**(원본) 내용.

### 3.4 해결 방법

- **내 내용만 남기기**: `<<<<<<<`, `=======`, `>>>>>>>` 줄 전체를 지우고, 남기고 싶은 본문만 둠.
- **원본 내용만 남기기**: 위와 같이 표시를 지우고, 원본 쪽 본문만 둠.
- **둘 다 살리기**: 표시를 지운 뒤, 두 내용을 합쳐서 한 덩어리로 정리함.

모든 충돌 파일에서 위 작업을 한 뒤 저장함.

### 3.5 merge 마무리하기

```bash
git add .
git commit -m "merge: upstream 변경 사항 반영, 충돌 해결"
git push origin main
```

커밋 메시지는 프로젝트의 커밋 규칙(예: `merge:` 또는 `chore:`)에 맞게 쓰면 됨.

## 4. Pull Request 할 때 주의할 점 (base 브랜치)

**Pull Request(PR)** 는 "내 변경 사항을 어떤 저장소의 어떤 브랜치에 반영해 달라"고 요청하는 것.

### 4.1 반드시 확인할 것

- **base 브랜치(base repository)**  
  PR을 만들 때 GitHub이 "어디에 합칠까?"라고 보여주는 **대상 저장소 + 브랜치**가 있음.
- **내 포크 → 원본**으로 PR을 보낼 때만 base를 **원본 저장소의 브랜치**(예: `owner/FoliumOnline`, `develop`)로 두어야 함.
- **원본 저장소 쪽에서 base를 "내 포크"로 두면 안 됨.**  
  즉, 원본 저장소에 "원본을 내 포크에 합치기" 같은 PR을 만들면 잘못된 것. 실수로 만들었다면 해당 PR은 닫음.

### 4.2 정리

- **원본에 기여할 때**: base = **원본 저장소**의 `main` 또는 `develop`, compare = **내 포크**의 해당 브랜치.
- **원본을 내 포크에 가져오는 일**은 보통 PR이 아니라, [2장](#2-원본origin-변경-사항을-내-포크에-가져오기)처럼 로컬에서 `git fetch` + `git merge` 한 뒤 `git push origin main` 으로 함.

## 5. 개발 서버 켜고 사이트 보기

내용을 수정한 뒤 화면에서 확인하려면 **개발 서버**를 켜야 함.

1. **Node.js**가 설치되어 있어야 함. (LTS 버전 권장.)
2. 프로젝트 폴더에서 터미널을 열고 다음을 **처음 한 번만** 실행함:
    ```bash
    pnpm install
    ```
    (없으면 `npm install -g pnpm` 으로 pnpm 설치 후 다시 실행.)
3. 개발 서버 실행:
    ```bash
    pnpm dev
    ```
4. 브라우저에서 다음 주소로 접속함:
    - **사이트**: http://localhost:4321 (또는 터미널에 안내된 주소)
    - **Keystatic(블로그 편집)**: http://127.0.0.1:4321/keystatic
    - **Admin(About 편집)**: http://127.0.0.1:4321/admin

서버를 끄려면 터미널에서 `Ctrl + C` 누름.

## 6. About me(소개) 페이지 채우기 — Admin 페이지

About me 페이지에 나오는 **프로필 사진, 이름, 소개 문구, 연락처, 경험/역량** 등은 **Admin 페이지**에서 폼으로 채울 수 있음.

### 6.1 Admin 페이지 열기

1. `pnpm dev` 로 개발 서버를 켠 상태에서 브라우저로 **http://127.0.0.1:4321/admin** 접속.
2. "About 관리" 페이지가 보이면 됨.

### 6.2 폼에 입력하기

- **프로필 사진**: URL을 넣거나, 프로젝트 안 이미지 경로(예: `/images/내사진.jpg`) 입력.
- **이름, 소개, 연락처(이메일, GitHub, LinkedIn)** 등 표시된 입력칸에 원하는 내용 입력.
- **경험/역량** 같은 섹션은 항목을 추가·수정·삭제할 수 있음. 빈 항목은 사이트에 안 나오므로 필요 없는 칸은 비워 두면 됨.

### 6.3 저장 후 사이트에 반영하기

- 폼에서 **저장** 또는 **다운로드** 버튼을 누르면 **JSON 파일**이 내려받아짐.
- 이 파일 내용을 프로젝트 안 **`src/data/about.json`** 파일로 교체해야 함.
    - `src/data/about.json` 파일을 연 다음, 다운로드한 JSON 내용 전체를 복사해서 붙여넣고 저장함.
- 저장 후 브라우저에서 **About me** 페이지(http://127.0.0.1:4321/about)를 새로고침하면 변경 내용이 보임.

Admin 페이지는 **개발 서버(`pnpm dev`)에서만** 동작함. 배포된 사이트에는 Admin 주소가 없을 수 있음.

## 7. 이력서(Resume) 채우기

사이트의 **이력서(Resume)** 페이지는 **JSON 파일**로 내용이 정해짐.

### 7.1 수정할 파일

- **한국어 이력서**: `src/data/resume.json`
- **영문 이력서**: `src/data/resume_en.json`

두 파일 구조는 같고, 언어만 다르게 채우면 됨.

### 7.2 어떤 내용을 넣는지

- **basics**: 이름, 직무 라벨, 프로필 사진 경로, 이메일, 전화번호, URL, 한 줄 요약, 주소, SNS(GitHub 등).
- **work**: 경력 (회사명, 직책, 기간, 요약, 하이라이트).  
  노출 분야를 쓰려면 `jobField`에 `"web"` 또는 `"game"`을 넣음.
- **projects**: 프로젝트 (이름, 기간, 설명, 하이라이트, URL, `jobField` 등).
- **education**: 학력 (학교, 전공, 학위, 기간, 과목 등).
- **skills**, **languages**, **awards**, **certificates** 등 필요한 항목만 채우면 됨.

JSON 형식이므로 **쉼표, 따옴표, 중괄호/대괄호**를 틀리지 않게 해야 함. 저장 후 `pnpm dev` 로 실행 중인 페이지에서 Resume를 새로고침해서 확인함.

## 8. 포트폴리오(Portfolio) 채우기

포트폴리오 **프로젝트 하나**는 **파일 하나**로 관리됨. 확장자는 `.mdoc`이고, **파일 이름**이 곧 주소의 **슬러그**가 됨.  
(예: `my-project.mdoc` → 사이트에서 `/portfolio/my-project` 로 보임.)

### 8.1 파일 위치

- 모든 포트폴리오 파일은 **`src/content/portfolio/`** 폴더 안에 있음.
- 새 프로젝트를 추가하려면 이 폴더에 **`원하는슬러그.mdoc`** 이름으로 새 파일을 만들면 됨.

### 8.2 파일 구조 (처음 부분 — 프론트매터)

파일 맨 위에는 **YAML 형태의 메타데이터**가 있음. `---` 로 시작하고 `---` 로 끝나는 블록.  
아래 필드를 채우면 됨.

| 필드                | 설명                             | 예시                        |
| ------------------- | -------------------------------- | --------------------------- |
| title               | 프로젝트 제목                    | `title: 내 웹 프로젝트`     |
| description         | 짧은 설명 (목록/카드에 보임)     | 한 줄 요약                  |
| startDate / endDate | 진행 기간 (YYYY-MM-DD)           | `"2024-01-01"`              |
| goal                | 프로젝트 목표                    | 한두 문장                   |
| role                | 내 역할                          | 한 줄                       |
| teamSize            | 참여 인원 수                     | `1` 또는 `3`                |
| accomplishments     | 성과 목록 (불릿)                 | `- 항목1` 다음 줄 `- 항목2` |
| keywords            | 기술/키워드 목록                 | `- JavaScript` 등           |
| github              | GitHub 저장소 주소 (없으면 `""`) | URL 또는 `""`               |
| public              | 목록에 노출 여부                 | `true` 또는 `false`         |
| jobField            | 노출 분야 (선택)                 | `web` 또는 `game`           |

날짜는 반드시 **따옴표**로 감싸는 것이 안전함. 예: `startDate: "2024-06-01"`.

### 8.3 본문 쓰기

`---` 로 끝나는 메타데이터 블록 아래부터는 **본문**.  
일반적인 문서처럼 제목(`## 제목`), 단락, 목록(`- 항목`), 링크 등을 쓰면 됨.  
이 부분이 포트폴리오 **상세 페이지** 아래쪽에 그대로 렌더링됨.

### 8.4 예시 (앞부분만)

```yaml
---
title: 나의 첫 웹 프로젝트
description: 회원가입과 게시판이 있는 웹 사이트를 만듦.
startDate: "2024-03-01"
endDate: "2024-06-01"
goal: 백엔드와 DB를 활용한 풀스택 경험을 쌓기 위함.
role: 프론트엔드 및 API 설계
teamSize: 2
accomplishments:
    - 반응형 UI 구현
    - REST API 설계 및 연동
keywords:
    - HTML
    - CSS
    - JavaScript
github: "https://github.com/내계정/프로젝트"
public: true
jobField: web
---
## 개요

이 프로젝트에서는 ...
```

이렇게 저장한 뒤 개발 서버가 켜져 있으면 **Portfolio** 목록과 **/portfolio/파일이름** 상세 페이지에서 확인할 수 있음.

## 9. 블로그 글 쓰기 — Keystatic

**블로그 포스트**는 코드나 JSON을 직접 건드리지 않고 **Keystatic**이라는 편집 화면에서 작성함.

### 9.1 Keystatic 열기

1. `pnpm dev` 로 개발 서버 실행.
2. 브라우저에서 **http://127.0.0.1:4321/keystatic** 접속.

### 9.2 글 쓰기 흐름

- Keystatic 화면에서 **콘텐츠 종류**(예: Posts/블로그 포스트)를 선택함.
- **새 포스트 추가** 또는 기존 포스트를 골라서 제목, 요약, 날짜, 카테고리, 태그, 본문 등을 입력함.
- 본문은 **마크다운**으로 쓸 수 있음. (제목, 굵게, 링크, 목록 등.)
- 이미지를 넣을 수 있는 항목이 있으면, 안내에 따라 업로드하거나 경로를 지정함.

### 9.3 저장과 반영

- Keystatic에서 **저장**하면 내용이 프로젝트 폴더의 **콘텐츠 파일**로 저장됨.
- 저장 후 블로그 목록/상세 페이지를 새로고침하면 글이 보임.

Keystatic은 **개발 서버가 돌아가는 동안**에만 사용 가능함. 배포된 사이트에는 Keystatic 주소가 없을 수 있음.

| 하고 싶은 일          | 어디서 / 어떤 파일                                                        |
| --------------------- | ------------------------------------------------------------------------- |
| 소개(About me) 채우기 | Admin 페이지 → JSON 다운로드 → `src/data/about.json` 에 덮어쓰기          |
| 이력서 채우기         | `src/data/resume.json`, `src/data/resume_en.json` 편집                    |
| 포트폴리오 추가·수정  | `src/content/portfolio/` 안에 `슬러그.mdoc` 파일 추가·편집                |
| 블로그 글 쓰기        | Keystatic (http://127.0.0.1:4321/keystatic)                               |
| 원본 변경 가져오기    | `git fetch upstream` → `git merge upstream/main` → `git push origin main` |
| PR 보낼 때            | base를 **원본 저장소**로 두고, 원본을 base로 하는 PR은 만들지 않기        |

추가로 궁금한 점이 있으면 프로젝트의 README.md, CHANGES.md, docs/ 폴더 안 문서를 참고하면 됨.
